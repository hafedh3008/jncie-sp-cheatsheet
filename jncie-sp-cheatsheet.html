IS-IS — Adjacences & Troubleshooting

▶
family iso NET unique MTU L1/L2 overload BFD

📋 Concepts clés

  • NET = 49.AREA.SYSTEM-ID.00 — system-id unique par routeur
  • family iso requise sur l'interface ET sur lo0.0 pour activer IS-IS
  • Niveaux : L1=intra-area, L2=inter-area, AS 65001 typique = L2 only
  • Métriques : wide=jusqu'à 16.7M, narrow=63 max (legacy)
  • Adjacence : NEIGHBOR_DOWN → INITIALIZING → UP

✅ Vérifications systématiques

show isis adjacency
show isis interface
show isis database
show route 10.220.1.0/24 protocol isis | match /32
show isis statistics

⚠️ Pièges identifiés lab

Piège 1 — family iso oublié
RR1 em2 avait family inet mais PAS family iso → 0 adjacence.
Diagnostic : adjacence absente même avec interfaces UP.
Piège 2 — NET dupliqué
PE7 utilisait NET de P4 (...1004.00) → adjacences instables.
Fix : rename ...1004.00 to ...1007.00
Piège 3 — MTU mismatch
P5 avait MTU 1500 hardcodé → IS-IS PDU LSP rejetés (>1492 octets).
Fix : delete interfaces ge-X mtu 1500 (revient à MTU défaut 1514).

💡 Tricks essentiels

Catch-all "toutes adjacences UP" = trouver TOUS les blockers, pas juste ceux du PDF. La solution officielle = "principaux fixes" — pas exhaustive.
Vérification cascade : si adjacences IS-IS down → BGP ne s'établit pas → L3VPN/EVPN HS → cascade de pertes (20+ pts).
passive sur lo0.0 = annonce le préfixe sans former d'adjacence (pas de hellos sur lo0).

🔧 Features avancées

# Overload bit (pas de transit X secondes au boot)
set protocols isis overload timeout 600

# BFD sub-second sans toucher hellos
set protocols isis interface ge-X.0 bfd-liveness-detection minimum-interval 500

# LDP synchronization (attendre LDP avant d'utiliser le lien)
set protocols isis interface ge-X.0 ldp-synchronization

# No IPv6 routing dans IS-IS (cas 6PE)
set protocols isis no-ipv6-routing

# Export policy (annoncer uniquement /32)
set protocols isis export isis-export

BGP — Diagnostic States (LE TRICK ESSENTIEL)

▶
Idle Connect Active Established

💡 Lecture des états BGP — diagnostic immédiat

Idle → BGP désactivé OU pas de route vers neighbor
Connect → BGP local tente OPEN, attente du SYN-ACK
→ Check : route vers neighbor, source-address, filtre lo0 (RE-protect)
Active → TCP s'établit mais OPEN pas validé
→ Check : ASN match, multihop si requis, TCP-AO keys
Established → session OK, échange UPDATES

✅ Diagnostic flow BGP

1. show bgp summary                    # état global
2. show bgp neighbor X.X.X.X           # détails session
3. ping X.X.X.X source LOCAL           # L3 connectivity
4. show route X.X.X.X                  # IGP résolution
5. show route table inet.0 X.X.X.X     # routing vers neighbor
6. show route table inet.3 X.X.X.X     # MPLS pour iBGP avec NHS
7. show route table bgp.l3vpn.0 hidden # routes hidden (NH unresolvable)

⚠️ Pièges Connect / Active fréquents

Filtre RE-protect bloque BGP
S1.1 utilise prefix-list BGP-Allow-VRF qui peut être vide.
Fix : set policy-options prefix-list BGP-Allow-VRF apply-path "routing-instances <*> protocols bgp group <*> neighbor <*>"
Layer 2 cassé en EVE-NG
Interface UP/UP mais ping fail, ARP vide → câblage EVE-NG sur mauvais port.
vMX mapping : em2=ge-0/0/0, em4=ge-0/0/2, em6=ge-0/0/4
AS mismatch
peer-as côté A ≠ ASN configuré côté B → Active.
Exemple lab : PE6 baseline peer-as 64588 mais CE8 = AS 64533.

iBGP / Route Reflectors

▶
cluster-id add-path resolution-ribs allow families

📋 Concepts clés

  • Cluster-ID = identifiant RR pour détection loops (ORIGINATOR_ID + CLUSTER_LIST)
  • allow X.X.X.0/24 = peering dynamique (clients dans subnet)
  • RR sans MPLS = doit résoudre next-hops VPN — 2 méthodes (legacy / moderne)
  • add-path RFC 7911 = multiple chemins par préfixe

✅ Vérifications RR

show bgp summary
show bgp neighbor X.X.X.X | match "Type|Cluster|Add Path"
show route table bgp.l3vpn.0
show route table bgp.evpn.0
show route advertising-protocol bgp X.X.X.X
show route receive-protocol bgp X.X.X.X

💡 RR sans MPLS — Legacy vs Moderne

Legacy (hack) : 
set routing-options rib inet.3 static route 0/0 discard
Fonctionne mais pas RFC. Crée artificiellement une "route MPLS".
Moderne (S2.8) : 
set routing-options resolution rib bgp.l3vpn.0 resolution-ribs inet.0
set routing-options resolution rib bgp.evpn.0 resolution-ribs inet.0
set routing-options resolution rib bgp.mvpn.0 resolution-ribs inet.0
Propre. Dit explicitement de résoudre via inet.0.

# Côté RR

set protocols bgp group rrc neighbor X.X.X.X family inet unicast add-path send path-count 2

# Côté client (réciproque)
set protocols bgp group rr neighbor X.X.X.X family inet unicast add-path receive
set protocols bgp group rr neighbor X.X.X.X family inet unicast add-path send path-count 2

eBGP & Policies

▶
prefix-limit communities as-override replace pattern local-pref

📋 Best-path BGP (ordre)

  • 1. Weight (Cisco) / Local-Pref+ (Junos)
  • 2. AS_PATH shorter
  • 3. Origin IGP < EGP < Incomplete
  • 4. MED lower
  • 5. eBGP > iBGP
  • 6. IGP metric to next-hop
  • 7. Router-ID lower

💡 Tricks eBGP

prefix-limit teardown (S2.6) — ferme la session au dépassement : 
set protocols bgp group transit family inet unicast prefix-limit maximum 1000
set protocols bgp group transit family inet unicast prefix-limit teardown
as-override (S2.9) — résout les boucles d'AS pour CEs partageant le même AS : 
set protocols bgp group CEX as-override
replace pattern (S2.12) — rename une community dans une policy : 
edit policy-options policy-statement REJECT_BAD_COMMUNITIES
replace pattern BAD_COMMUNITIES with FROM_DATA_CENTER
Exit point forcé (S2.10) — augmenter local-pref : 
set policy-options policy-statement nhs term 1 then local-preference 200

🔧 Routes hidden — diagnostic

# Cherche : "Unusable" "rejected by import" "AS loop" "MED inferior"

# Causes :
# 1. Next-hop unresolvable → S2.8 (resolution-ribs)
# 2. Import policy reject → adapter policy
# 3. AS-PATH loop → as-override ou as-loops N
# 4. MED inferior → comportement normal

TCP-AO (Authentification BGP)

▶
aes-128-cmac-96 send-id recv-id keychain

⚠️ POINT CRITIQUE — send-id / recv-id

Asymétriques entre les 2 routeurs !
RR1 : send-id=3, recv-id=8
PE3 : send-id=8, recv-id=3 (INVERSÉS)
Rule : send-id côté A = recv-id côté B.
Si pas symétrique → session jamais Established.

set security authentication-key-chains key-chain IBGP-AUTH key 0 ao-attribute tcp-ao-option enabled
set security authentication-key-chains key-chain IBGP-AUTH key 0 ao-attribute cryptographic-algorithm aes-128-cmac-96
set protocols bgp group rrc neighbor X.X.X.X authentication-key-chain IBGP-AUTH

RSVP-TE / MPLS

▶

📋 Concepts clés RSVP-TE

  • LSP = Label Switched Path — chemin MPLS avec réservation BW possible
  • CSPF = Constrained SPF — calcule chemin selon contraintes
  • admin-groups = "couleurs" des liens (BLUE, GREEN, etc.)
  • FRR link-protection vs node-link-protection (NLP recommandé)
  • inet.3 = table des LSPs résolus pour next-hop BGP

💡 Tricks RSVP

admin-groups (S2.14) — 3 variantes : 
include-any BLUE    # au moins un lien BLUE
include-all "B G"   # ET BLUE ET GREEN
exclude RED         # bannir RED
Path strict (S3.2) — forcer hop-by-hop :

set protocols mpls path BLUE-PATH 10.220.1.5 strict
set protocols mpls path BLUE-PATH 10.220.1.3 strict
node-link-protection (S2.15) — protège lien + nœud aval :

LDP — Sync & Downstream

▶
ldp-synchronization egress-policy FEC

📋 Concepts LDP

  • LDP distribue labels basé sur IGP (next-hop = same FEC)
  • Par défaut, LSR annonce seulement son propre lo0 en FEC
  • egress-policy permet d'annoncer d'autres loopbacks

💡 IS-IS LDP synchronization (S2.13)

Sans sync : link UP → IS-IS converge en 6s, LDP en 30s → trou MPLS 30s.
Avec sync : IS-IS annonce métrique infinie tant que LDP pas prêt → traffic reste sur backup.

# Optionnel : hold-time pour cas LDP foire
set protocols isis interface ge-X.0 ldp-synchronization hold-time 60

L3VPN — VRF Hub-and-Spoke

▶
RD RT vrf-target vrf-import/export vrf-table-label

📋 Concepts L3VPN

  • RD (Route Distinguisher) — 8 octets, rend les VPN routes uniques
  • RT (Route Target) — community étendue, contrôle import/export
  • vrf-target = shortcut pour RT import+export même valeur
  • vrf-import/export = policies fines (cas H&S)
  • vrf-table-label = IP-lookup sur le label VPN reçu

💡 Hub-and-Spoke avec 2 RT (S3.3)

RT 1:1 = Spoke→Hub
RT 1:2 = Hub→Spoke
Spokes : import target:1:2, export target:1:1 (importent du Hub, exportent au Hub)
Hub : import target:1:1, export target:1:2 (importe des Spokes, exporte aux Spokes)
Aucun spoke n'importe 1:1 → ils ne se voient pas entre eux

6PE — IPv6 over MPLS IPv4

▶
ipv6-tunneling ::ffff: inet6.3 labeled-unicast RFC 4798

📋 Les 3 piliers du 6PE

  • 1. mpls ipv6-tunneling — crée la table inet6.3 (lookup IPv6 → label)
  • 2. LSP RSVP IPv4 entre les 2 PE (transporte les paquets IPv6)
  • 3. Next-hop IPv4-mapped IPv6 (::ffff:<ipv4>) — résout via inet.3

🔧 Config 6PE (S2.9)

# Sur les 2 PE (PE2 et PE6 dans le lab)
set protocols mpls ipv6-tunneling

# Policy export-rr pour next-hop IPv4-mapped IPv6
set policy-options policy-statement export-rr term ipv6 from family inet6
set policy-options policy-statement export-rr term ipv6 then next-hop ::ffff:10.220.1.X
set policy-options policy-statement export-rr term ipv6 then accept

# LSP RSVP entre les 2 PE
set protocols mpls label-switched-path TO-PEX to 10.220.1.X
set protocols mpls label-switched-path TO-PEX primary TO-PEX

# Family BGP labeled-unicast
set protocols bgp group rr family inet6 labeled-unicast explicit-null

⚠️ Pièges 6PE du lab

Piège AS : PE6 baseline avait peer-as 64588 mais CE8 = AS 64533.
Fix : replace pattern 64588 with 64533
Piège subnet IPv6 : PE6 avait 2001:db8:104:1::1/127 mais CE8 = 2001:db8:104::/127.
Fix : replace pattern 2001:db8:104:1::1/127 with 2001:db8:104::1/127
as-override requis si CE5 et CE8 partagent le même AS → sinon AS-loop detection drop.

CoS — Pipeline complet

▶
classify queue schedule rewrite policer

📋 Pipeline CoS

  • Classify → BA (DSCP/EXP) ou MF (firewall filter) → forwarding-class
  • Queue → 4 queues default (best-effort, expedited, assured, network-control)
  • Schedule → priority (strict-high, high, medium-high, low)
  • Rewrite → réécriture DSCP / EXP en sortie

🔧 MF Classifier (S2.18)

set firewall family inet filter CE11-VOICE term VOICE from source-address 192.168.113.1/32
set firewall family inet filter CE11-VOICE term VOICE then forwarding-class voice
set firewall family inet filter CE11-VOICE term VOICE then accept
set firewall family inet filter CE11-VOICE term DEFAULT then accept
set interfaces ge-X unit 0 family inet filter input CE11-VOICE

🔧 Scheduler + Map (S2.19)

set class-of-service schedulers voice transmit-rate percent 10
set class-of-service schedulers voice buffer-size percent 10
set class-of-service schedulers voice priority strict-high

set class-of-service scheduler-maps MAP forwarding-class voice scheduler voice
# ... autres FC

set class-of-service interfaces ge-X scheduler-map MAP

🔧 EXP Rewrite (S2.20)

# Marquer voice → EXP 010 (=2)
set class-of-service rewrite-rules exp voice-rewrite forwarding-class voice loss-priority low code-point 010
set class-of-service interfaces ge-X unit 0 rewrite-rules exp voice-rewrite

✅ Vérifications CoS

show class-of-service interface ge-X
show class-of-service scheduler-map MAP
show interfaces queue ge-X

💡 Priority strict-high vs high

strict-high = absolument prioritaire (peut starver les autres si pas de rate-limit)
high = prioritaire mais cohabite (weighted)
Voice prioritized "before others" → utiliser strict-high + transmit-rate

RE-Protect / Firewall Filters

▶
lo0 filter apply-path prefix-list term ordering

📋 Concepts RE-Protect

  • Filter family inet appliqué input sur lo0.0 → protège le RE
  • Ordre des terms IMPORTANT (séquentiel, premier match gagne)
  • Term final = DEFAULT discard (avec syslog)
  • Prefix-list avec apply-path = auto-population dynamique

⚠️ Piège critique BGP-Allow-VRF vide

S1.1 utilise source-prefix-list BGP-Allow-VRF dans le term bgp-allow.
Si BGP-Allow-VRF n'a pas d'apply-path → vide → BGP VRF bloqué !
Fix : 
set policy-options prefix-list BGP-Allow-VRF apply-path "routing-instances <*> protocols bgp group <*> neighbor <*>"

# Insère gRPC AVANT DEFAULT (S1.3)
insert firewall family inet filter re-protect term gRPC before term DEFAULT

✅ Vérifications filtre

show configuration firewall family inet filter re-protect
show firewall filter re-protect
show firewall counter re-protect
show policy-options prefix-list BGP-Allow
show policy-options prefix-list BGP-Allow-VRF

EVE-NG — Pièges & Mapping vMX

▶
em mapping paste corruption paste delay SCP load

📋 vMX EVE-NG mapping (CRITIQUE)

  • em0 = fxp0 (mgmt)
  • em1 = internal (pas utilisé)
  • em2 = ge-0/0/0 (premier dataport)
  • em3 = ge-0/0/1
  • em4 = ge-0/0/2
  • em5 = ge-0/0/3
  • em6 = ge-0/0/4
  • em7 = ge-0/0/5

⚠️ PIÈGE Layer 2 cassé EVE-NG

Symptôme : interface UP/UP mais ping fail, ARP vide.
Cause : config sur ge-0/0/2 mais EVE-NG câble sur em6 (= ge-0/0/4).
Diagnostic : hover sur le lien EVE-NG → voir le mapping em ↔ ge.
Fix : adapter la config à la VRAIE interface câblée.

⚠️ PIÈGE Paste corruption vMX

Symptôme : interface → interset, hashes password tronqués, "syntax error" sur lignes spécifiques.
Cause : buffer série vMX EVE-NG dépassé.
Fix : Paste delay SecureCRT : 
Options → Session Options → Terminal → Emulation
  Line send delay : 100 ms
  Character send delay : 10 ms

💡 Workflow push config par blocs

1. Configure paste delay dans SecureCRT (Line 100ms, Char 10ms)
2. Split fichier en blocs ≤ 18 lignes (testé safe)
3. PowerShell pour préparer clipboard : 
$lines = Get-Content "X.txt" | Where-Object { $_ -match '^\s*set\s' }
$lines[0..17] -join "`n" | Set-Clipboard
4. Sur Junos : load set terminal + paste + Ctrl+D
5. commit check entre chaque bloc → détecte la corruption tôt

💡 Alternative SCP + load set

Si paste impossible/risqué : 
# Sur Junos
file copy scp://user@host/path/file.conf /var/tmp/file.conf
load set /var/tmp/file.conf

Plus fiable que le paste interactif.
